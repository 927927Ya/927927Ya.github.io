2. 整体架构

为什么不在 Dify 里直接处理？

Dify 的沙箱（gVisor）有几条硬限制，导致视频处理走不通：

限制 影响
禁止 subprocess 无法运行 FFmpeg
禁止写临时文件 tempfile 权限不足
变量 200KB 上限 base64 图片会被截断
迭代节点变量作用域隔离 迭代内部结果外部拿不到

解决思路： 把「视频下载 + FFmpeg 处理 + AI 分析 + 飞书写入」全部抽到一个独立的 Docker 容器（ffmpeg-server）中，Dify 只负责发起 HTTP 调用和展示结果。

6. 第四步：在 Dify 中搭建工作流

6.0 认识 DSL 文件

本教程提供的 视频反推分镜Agent.yml 是一个 Dify DSL (Domain Specific Language) 文件，它将整个工作流的节点配置、连线关系、变量定义全部导出为一个 YAML 格式的文本文件。DSL 文件的作用是：

  • 一键还原工作流：无需手动拖拽，导入后自动生成所有节点
  • 版本管理：可以像代码一样用 Git 管理工作流版本
  • 跨实例迁移：从一个 Dify 实例导出，导入另一个实例

变量引用说明：输入变量 http_body 的值来自上一个节点「提取视频关键帧」返回的 body。在 Dify 中，HTTP 请求节点返回的完整响应内容通过 {{#节点名.body#}} 引用。

作用 将每个分镜的关键帧截图发给通义千问视觉模型，AI 逐帧分析并生成结构化描述（画面主体、动作、环境、运镜、风格）

在 Dify 界面中操作：

为什么超时设 600 秒：如果视频有 30 个分镜，每个分镜调一次千问 API（约 2-3 秒），加上 API 调用间隔（0.5 秒防限流），总耗时可能达到 90 秒以上。更大的视频可能更久。

核心原理：ffmpeg-server 收到请求后，对 shots 数组中的每个分镜：

  1. 读取对应的关键帧 JPEG 图片
  2. 转为 base64
  3. 拼装成多模态 API 请求体（图片 + 提示词）
  4. 发送到千问 API
  5. 解析返回的结构化 JSON（subject、action、environment、camera_movement 等）
  6. 所有分镜处理完后返回完整结果

  2. 点击「运行」

项目 建议值 说明

7.2 观察执行

工作流最耗时的环节是：

  • 提取视频关键帧（FFmpeg 场景检测，几秒到几十秒）
  • 生成分镜脚本（逐帧调千问 API，每个分镜约 2-3 秒，12 个分镜约 30 秒）
  • 同步图片到飞书（上传 12 张图片，约 20-40 秒）

总耗时通常在 1-2 分钟。

8. 第六步：常见问题排查

Q1：HTTP 节点报错「Reached maximum retries」

原因：视频太长或分镜太多，处理时间超过超时设置。

解决：

  • 增加 HTTP 节点「最大读取超时」到 600 秒以上
  • 或者上传更短的视频

Q2：飞书记录重复（24 条而不是 12 条）

原因：Dify 底层在网络波动时会隐式重试 HTTP 请求。

解决：确保「飞书创建记录」节点调的是 http://ffmpeg-server:5002/feishu-create（带查询去重），而不是直接调飞书 API。

Q3：所有分镜显示「API错误:401」

原因：通义千问 API Key 无效或过期。

解决：

  1. 登录 dashscope 检查 Key 是否有效
  2. 更新 docker-compose.yaml 中的 LLM_API_KEY
  3. 重建容器：docker compose up -d --build ffmpeg-server

Q4：图片在 Dify 输出中不显示

原因：image_url 是相对路径，浏览器无法解析。

解决：确认 nginx 模板已添加 /shots 代理并重启 nginx。Dify 输出中的图片路径应为 /shots/xxx.jpg，通过 nginx 代理访问。

Q5：ffmpeg-server 日志显示「下载视频失败: HTTP 404」

原因：Dify 文件签名 URL 在容器间无法验证。

解决：

  1. 确认 .env 中 STORAGE_TYPE=local
  2. 确认 docker-compose 中 ffmpeg-server 挂载了 ./volumes/app/storage:/app/storage:ro
  3. 确认 server.py 中使用 http://api:5001 下载文件（而不是 http://nginx）
  4. 重启 Dify：docker compose down && docker compose up -d

Q7：想清空飞书表格重新测试

解决：

  1. 进入飞书多维表格 → 选中所有记录 → 右键「删除记录」
  2. 或使用飞书 API 批量删除（获取 record_id 后调 batch_delete）
  3. 删除后重新运行 Dify 工作流即可

因为 /feishu-create 有查询去重，旧记录不删完会导致新视频的分镜被误判为”已存在”而跳过创建。每次测试新视频前建议清空表格。
