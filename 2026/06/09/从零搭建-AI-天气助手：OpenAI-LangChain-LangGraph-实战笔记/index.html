二、LangGraph 方式：声明式流程图

当任务变复杂（多轮工具调用、多 Agent 协作、人机确认），手动写 if/else 会越来越乱。LangGraph 用**图（Graph）**的方式把流程声明出来。

2.1 核心概念

概念 说明
State 所有节点共享的”公共黑板”，节点可以读取和修改
Node 一个处理步骤，就是一个普通函数
Edge 节点之间的连线，数据流动方向
条件边 根据状态动态决定下一步走向

三、LangChain 流水线 vs LangGraph 流程图

对比项 LangChain 流水线 LangGraph 流程图
结构 线性串联 A → B → C 节点+边构成图，可分叉、可循环
路径 编码时固定 运行时动态决定（条件边）
状态 隐式传递，组件间简单传参 显式 State 对象，所有节点共享可读写
循环 不支持（需手动套循环） 原生支持循环
适用场景 翻译、摘要等固定步骤任务 多 Agent 协作、需要判断分支的复杂工作流

3.1 如何选择？

  • 步骤固定、单向流动 → 用 LangChain Pipeline（.pipe() / RunnableSequence）
  • 需要判断、循环、多 Agent 协作 → 用 LangGraph（StateGraph）

五、常见问题

Q1: 为什么用 DynamicStructuredTool 而不是 Tool？

DynamicStructuredTool 支持用 zod 精确定义参数结构，LLM 生成的参数更准确，且自带类型校验。

Q2: Annotation 和旧版 channels 写法有什么区别？

Annotation 是 LangGraph 新版本推荐写法，类型推断更好、语法更现代：

  • reducer 替代 value
  • Annotation.Root({...}) 替代 { channels: {...} }

Q3: 聚合数据天气 API 免费吗？

有免费额度，注册后每天 50 次调用，学习测试完全够用。

Q4: 消息数统计有什么用？

帮助理解 LLM 和工具之间的交互轮次，调试时看流程是否正常运转。

六、总结

文件 学到的内容
01.mjs 最原始的 OpenAI Function Calling，手动拼消息
02.mjs LangChain 的 PromptTemplate、Chain、Memory
03.mjs LangChain 的 Tool + bindTools()，手动控制调用流程
04.mjs LangGraph 的 StateGraph + Annotation，声明式管理复杂流程

从手动控制到声明式编排，代码越来越简洁，逻辑越来越清晰。当 Agent 需要多轮思考、多工具协作、甚至多个 Agent 配合时，LangGraph 的优势会越来越明显。
