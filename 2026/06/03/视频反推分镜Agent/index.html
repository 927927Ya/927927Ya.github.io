从零搭建 Dify 视频反推分镜 Agent：一个 Fallback 驱动的工程实践

上传一段视频，自动拆分成多个分镜，每个分镜带关键帧、AI 描述，输出一份可直接用的分镜脚本。

背景

做视频创作时，经常需要把成品视频「反推」回分镜脚本——分析它的镜头切换、画面构成、运镜方式。人工做这事很费时，特别是长视频。

目标很直接：上传一个视频，自动拿到分镜脚本。技术选型上，团队已经在用 Dify（自部署 1.13.3），所以最好的方式就是基于 Dify 工作流来实现。

遇到的坑

做一个看似简单的功能，踩的坑比预期的多。总结一下有价值的问题：

1. Dify 沙箱 ≠ 通用 Python 环境

Dify 的 Code 节点跑在 gVisor 沙箱中，没有 subprocess、没有任意文件写入权限。这意味着任何需要调用系统工具（FFmpeg、ImageMagick 等）或写临时文件的操作都无法在 Code 节点中完成。

教训：不要把 Dify Code 节点当通用后端用——它更适合做轻量数据转换，不适合做计算密集型任务。

2. 迭代节点变量作用域

Dify 的迭代节点中产生的变量在迭代外部无法通过 {{节点路径}} 引用。如果你在迭代里处理每一帧然后把所有结果拼起来——外面拿不到。

解决：不要用 Dify 迭代做聚合型任务，要么在迭代内部完成所有输出（拼接 Markdown），要么把循环逻辑搬到外部服务。

3. Dify 变量大小限制

base64 编码的图片动辄几百 KB，Dify 变量有 200KB 限制，直接传图片变量会被静默截断。

解决：图片不经过 Dify 变量传递，在 ffmpeg-server 中直接调 LLM。

4. SSRF 代理导致容器间通信失败

Dify 的 HTTP 请求节点通过 SSRF 代理发出请求。如果目标服务在 Docker 内部网络中，但代理白名单里没有它，请求会被拦截。

解决：把服务名（ffmpeg-server）加到 SSRF 代理的 allowed_domains 中，并确保两个服务在同一网络上。

架构复盘

回头来看，为什么不用更流行的方案（比如直接写一个 Node.js/Python 脚本）？

因为需求不是「做一个视频处理工具」，而是「在 Dify 里增加一个视频处理能力」。

Dify 作为已有平台，提供了用户管理、文件上传、权限控制、日志追溯——这些是我不想重新实现的。代价是必须接受 Dify 的约束（沙箱限制、网络架构、变量模型），然后在这个约束下找到最简可行的路径。

最终的 Sidecar 模式其实很轻量：一个 Flask 文件 + 一个 Dockerfile，核心逻辑不到 200 行 Python。Dify 端只有 4 个节点的简单工作流。

最直接的路不一定是最短的，但往往是最终走通的那条。
